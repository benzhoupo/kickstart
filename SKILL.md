# kickstart -- 项目启动引导

从一个模糊的想法到一份可以直接开干的项目规格说明书。通过结构化问答帮用户理清需求、选定技术栈、设计架构、规划实施步骤，最终输出一份 Markdown 文档保存到用户工作目录。

## 三条核心原则

**原则一：问对的问题。** 自适应问答，不堆砌。根据项目类型动态调整后续问题。后端 API 项目不问 UI 框架，个人项目不问 CI/CD。每轮最多问 3 个问题。

**原则二：用户做主。** 所有技术决策（编程语言、框架、数据库、UI 方案、部署方式等）都必须让用户自己选。使用 AskUserQuestion 工具列出选项 + 简短描述，不标"推荐"，不替用户做决定。用户主动说"你来定"时才给建议。

**原则三：输出可执行。** 生成的文档不是 PPT，是可以直接用来写代码的施工蓝图。每个部分具体到可以 `npm init` / `pip install` / `go mod init` 的程度。

## 问答流程

整个流程分为 5 个阶段，按顺序执行。完整的分支逻辑和跳过条件见 `references/qa-flow.md`。

### Phase 1: 核心需求（必问）

3 个必问问题，不可跳过：

1. **你想做什么？** -- 开放式提问，一两句话即可。从回答中提取关键词，初步判定项目类型。

2. **给谁用的？** -- 使用 AskUserQuestion，选项：个人项目 / 团队内部 / 面向用户的产品 / 开源项目。这决定了后续工程化决策的深度。

3. **核心功能是什么？** -- 让用户列 3-5 个关键词。从中识别特殊技术需求（实时通信、文件上传、支付、AI/ML 等）。

### Phase 2: 项目类型判定

根据 Phase 1 的回答自动判定项目类型：全栈 Web 应用 / 纯后端 API / 纯前端 / CLI 工具 / 移动端 / 库或SDK / 数据管线。

判定后告知用户："听起来你要做的是一个 {类型}，对吗？" 用户确认或修正。

各类型的详细画像（默认架构、问哪些问题、跳过哪些）见 `references/project-type-profiles.md`。

### Phase 3: 技术选型

根据项目类型，选择性地提问。所有问题使用 AskUserQuestion 工具，列出 2-4 个选项，每个选项附简短描述。

可能涉及的决策点：
- 编程语言
- 前端框架（全栈/纯前端项目）
- 后端框架（全栈/纯后端项目）
- 数据库（需要持久化的项目）
- ORM / 数据层
- CSS / UI 方案（有前端的项目）
- API 风格：REST / GraphQL（后端项目）
- CLI 框架（CLI 项目）
- 移动端框架：原生 / 跨平台（移动端项目）

选项来源见 `references/tech-stack-catalog.md`。只问当前项目类型需要的问题，其余跳过。

如果 Phase 1 中检测到特殊功能关键词（实时、文件上传、支付等），在此阶段追问具体方案。

### Phase 4: 工程化决策（按需）

根据项目类型和目标用户决定是否要问：
- 认证方案
- 部署方式
- CI/CD
- 测试策略
- 容器化
- 监控/日志

个人项目跳过全部。面向用户的产品全部要问。团队内部项目选择性地问。

开源项目额外问：License 类型、是否需要 CONTRIBUTING.md。

### Phase 5: 确认和生成

1. 以表格形式汇总所有决策，让用户确认或修改
2. 用户确认后，根据输出模板生成文档
3. 文件名：`{project-name}-spec.md`
4. 保存到用户当前工作目录
5. 告知用户文档路径，提示可以直接让 Claude Code 按规格开始编码

输出模板见 `references/output-template.md`。

## 问答行为规则

- 每轮使用 AskUserQuestion 工具最多问 3 个问题
- 所有选项平等展示，不标"推荐"
- 用户说"你来定""都行""随便" → 仅对当前问题给出一个建议并使用
- 用户说"跳过""差不多了""直接生成" → 剩余问题使用常见默认值，跳到 Phase 5
- 用户初始描述非常详细 → 自动提取信息，减少提问轮次
- 中文交互，技术名词保持英文（React、PostgreSQL、Docker 等不翻译）
- 不要翻译腔。用"前端你打算用什么？"而不是"请问您希望使用什么框架？"
- 先复述理解再提问："你想做一个 XX，那我问几个问题帮你定一下技术方案。"

## 输出格式

生成的 Markdown 文档包含 8 个 Section：

1. **项目概览** -- 名称、描述、项目类型、目标用户、核心功能列表
2. **技术栈** -- 表格，每项附简短说明（只包含用户选了的项）
3. **系统架构** -- 架构模式、核心模块、数据流描述
4. **目录结构** -- tree 格式，注释每个目录的用途，根据用户选的框架生成真实结构
5. **核心依赖** -- 生产依赖和开发依赖两张表（包名 + 用途）
6. **开发环境搭建** -- 前置要求 + 从零到 run dev 的完整步骤
7. **数据模型** -- 核心实体和关系（无数据库的项目省略此节）
8. **实施计划** -- 分 3-4 个阶段，每阶段列 checkbox 任务

架构模式的描述参考 `references/architecture-patterns.md`。

## 边界情况处理

- **用户已有项目** -- 如果检测到用户说"我已经有一个 XX 项目"，调整问法，只问增量部分
- **描述太模糊** -- 如"做个网站"，追问一个问题：这个网站是用来干什么的？
- **非常规技术栈** -- 接受用户的选择，如实记录到输出文件

## 搭配使用

生成规格文件后，用户可以直接对 Claude Code 说"按这个规格搭建项目"，Claude Code 会读取文件并开始创建项目结构、安装依赖、写基础代码。
