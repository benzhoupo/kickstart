# 问答流程定义

## Phase 1: 核心需求（必问，不可跳过）

### Q1.1: 项目描述
- 问法："你想做什么？一两句话说说就行。"
- 目的：确定项目大方向
- 解析规则：从回答中提取关键词，初步判定项目类型

### Q1.2: 目标用户
- 使用 AskUserQuestion 工具
- 问法："这个项目是给谁用的？"
- 选项：
  - 个人项目 -- 自己用，不考虑多用户
  - 团队内部 -- 公司或团队内部使用
  - 面向用户的产品 -- 需要上线给外部用户
  - 开源项目 -- 开源社区
- 影响：
  - 个人项目 → 跳过认证、CI/CD、监控相关问题
  - 面向用户 → 必须问认证、部署、监控
  - 开源 → 必须问文档、License、贡献指南

### Q1.3: 核心功能
- 问法："核心功能列一下，3-5 个关键词就行。"
- 目的：确定特殊技术需求
- 关键词触发：
  - "实时" → 后续追问 WebSocket/SSE
  - "文件上传" → 追问存储方案
  - "支付" → 追问支付渠道
  - "AI/ML" → 追问 ML 基础设施
  - "地图" → 追问地图 API

## Phase 2: 项目类型判定

### 自动判定规则

| 信号 | 判定为 |
|------|--------|
| 提到"页面""界面""前端" + "API""数据库" | 全栈 Web 应用 |
| 只提到"API""服务""接口" | 纯后端 API |
| 只提到"页面""组件""UI" | 纯前端 |
| 提到"命令行""CLI""脚本" | CLI 工具 |
| 提到"App""iOS""Android" | 移动端 |
| 提到"包""库""SDK" | 库/SDK |
| 提到"数据处理""ETL""pipeline" | 数据管线 |

### 确认
- "听起来你要做的是一个 {类型}，对吗？"
- 用户修正 → 重新判定

## Phase 3: 技术选型（分支逻辑）

**所有技术选型问题必须使用 AskUserQuestion 工具，列出选项让用户选，不标"推荐"。**

### 分支表

| 项目类型 | 要问的问题 | 跳过的问题 |
|---------|------------|-----------|
| 全栈 Web | 语言、前端框架、后端框架、数据库、ORM、CSS/UI 方案 | 无 |
| 纯后端 API | 语言、后端框架、数据库、ORM、API 风格 | 前端框架、CSS 方案 |
| 纯前端 | 前端框架、CSS/UI 方案、状态管理 | 后端、数据库、ORM |
| CLI 工具 | 语言、CLI 框架 | 前端、数据库 |
| 移动端 | 原生/跨平台、框架 | 后端单独处理 |
| 库/SDK | 语言、构建工具、发布方式 | 前端、数据库 |
| 数据管线 | 语言、数据存储、调度方式 | 前端 |

### 问题示例

**编程语言**（AskUserQuestion）：
- header: "编程语言"
- 选项根据项目类型动态调整，参考 `references/tech-stack-catalog.md`

**前端框架**（AskUserQuestion）：
- header: "前端框架"
- 选项：React / Vue 3 / Svelte / Angular
- 每个选项附简短描述

**后端框架**（AskUserQuestion）：
- header: "后端框架"
- 选项根据用户选的语言动态调整

**数据库**（AskUserQuestion）：
- header: "数据库"
- 选项：PostgreSQL / MySQL / MongoDB / SQLite

**CSS/UI 方案**（AskUserQuestion）：
- header: "CSS/UI"
- 选项：Tailwind CSS / CSS Modules / Ant Design / MUI

### 特殊功能追问

根据 Phase 1 关键词触发额外问题：

| 关键词 | 追问 |
|--------|------|
| "实时" | 实时方案：WebSocket / SSE / Socket.io |
| "文件上传" | 存储方案：本地 / S3 / OSS |
| "支付" | 支付渠道：微信支付 / 支付宝 / Stripe |

## Phase 4: 工程化决策（按需）

### 必要性矩阵

| 决策项 | 个人项目 | 团队内部 | 面向用户 | 开源 |
|--------|---------|---------|---------|------|
| 认证方案 | 跳过 | 简单方案 | 必问 | 按需 |
| 部署方式 | 跳过 | 必问 | 必问 | 文档说明 |
| CI/CD | 跳过 | 按需 | 必问 | 必问 |
| 测试策略 | 跳过 | 按需 | 必问 | 必问 |
| 容器化 | 跳过 | 按需 | 按需 | 按需 |
| 监控/日志 | 跳过 | 按需 | 必问 | 跳过 |

"必问"的决策项同样使用 AskUserQuestion 让用户选。

### 开源项目额外项
- License 类型：MIT / Apache 2.0 / GPL v3
- 是否需要 CONTRIBUTING.md 模板

## Phase 5: 确认和生成

### 确认汇总
以表格形式展示所有已收集的决策，让用户确认或修改。

### 生成流程
1. 读取 `references/output-template.md` 模板
2. 用收集到的信息填充模板
3. 文件名：`{project-name}-spec.md`（项目名 slugify 处理）
4. 保存到用户当前工作目录
5. 告知用户："文档已生成：{filename}。你可以直接让我按这个规格开始写代码。"

## 快捷行为

| 用户说 | 处理方式 |
|--------|---------|
| "你来定" / "都行" / "随便" | 对当前问题给出一个建议并使用 |
| "跳过" / "差不多了" / "直接生成" | 剩余问题使用常见默认值，跳到 Phase 5 |
| 用户初始描述非常详细 | 自动提取尽可能多的信息，减少提问轮次 |
