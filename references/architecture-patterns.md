# 架构模式参考

为输出文档的"系统架构"部分提供参考。根据用户选择的技术栈匹配合适的架构描述。

## 单体应用

- **适用**：MVP、小团队、简单业务逻辑
- **不适用**：多团队协作、需要独立扩缩容的场景
- **特征**：一个代码库，一个部署单元，所有功能在同一进程
- **典型栈**：Django / Rails / Spring Boot + 单数据库

## 前后端分离

- **适用**：中型项目、前后端不同开发节奏、需要多端（Web + 移动）
- **不适用**：极简项目（增加复杂度不值得）
- **特征**：前端 SPA 通过 API 与后端通信，各自独立部署
- **典型栈**：React/Vue + Express/FastAPI + PostgreSQL

## 全栈框架一体化

- **适用**：快速上线、SEO 重要、全栈 JS/TS 团队
- **不适用**：前后端团队分离、非 JS 后端
- **特征**：前端和后端在同一框架中，共享代码和类型
- **典型栈**：Next.js / Nuxt 3 / SvelteKit + Prisma/Drizzle + PostgreSQL

## 微服务

- **适用**：大型团队、复杂业务域、需要独立部署和扩缩容
- **不适用**：小团队、新项目起步、业务边界不清晰
- **特征**：多个独立服务，各自有数据库，通过 API 或消息队列通信
- **典型栈**：多语言 + Docker + Kubernetes + 消息队列

## Serverless

- **适用**：事件驱动、流量波动大、按调用计费
- **不适用**：长时间运行任务、需要 WebSocket 长连接、冷启动敏感
- **特征**：函数粒度部署，无需管理服务器
- **典型栈**：AWS Lambda / Cloudflare Workers / Vercel Functions

## Monorepo

- **适用**：前后端共享代码、多包项目、需要统一版本管理
- **不适用**：单一小项目、团队对工具链不熟悉
- **特征**：多个子项目在同一仓库，共享依赖和配置
- **工具**：Turborepo / Nx / pnpm workspace / Lerna
